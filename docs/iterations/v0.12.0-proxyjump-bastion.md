# Termex v0.12.0 — SSH ProxyJump 跳板机功能完整迭代

## 迭代概述

v0.12.0 实现生产级运维场景的核心需求：**通过跳板机（Bastion Host）安全访问内网服务器**。

- ✅ 单级 + 多级跳板机支持（无限级数）
- ✅ 跳板机 Session 共享（连接池）
- ✅ SSH Agent Forwarding（无需存储私钥）
- ✅ Sidebar 特殊渲染（徽章 + 引用计数）
- ✅ Tab 化 UI 设计（授权信息 + SSH 隧道分离）

---

## 背景与痛点

生产运维中的典型场景：

```
开发者笔记本
    ↓
公网跳板机（10.0.0.100）
    ↓
DMZ 中转机（172.16.0.50）
    ↓
内网数据库（192.168.1.10） — 无法直接访问，必须通过跳板机中转
```

**当前 Termex 的局限**：
- 只支持直连，无法通过跳板机访问内网
- 用户必须手动建立两个独立连接并切换窗口
- 跳板机配置变化时需要逐一修改所有内网服务器配置

---

## 核心设计

### 1. 外部跳板机连接池

跳板机 Session 独立管理，不嵌套在目标 Session 内，支持多个内网服务器复用同一跳板机连接：

```rust
// AppState 中的连接池
pub proxy_sessions: TokioRwLock<HashMap<String, ProxyEntry>>

// key = bastion server_id
// value = ProxyEntry { handle, ref_count }
```

**引用计数机制**：
- 内网服务器连接建立时，对跳板机 `ref_count += 1`
- 内网服务器断开时，对跳板机 `ref_count -= 1`，降为 0 时自动关闭跳板机连接

**单跳板机 10 个内网连接**：只需 1 个跳板机 Session（对比 10 个）

### 2. 多级跳板机支持

递归解析 `proxy_id` 形成连接链，从外到内逐层建立：

```
目标：internal-db（proxy_id = dmz-relay, dmz-relay.proxy_id = public-gw）
解析链：[internal-db, dmz-relay, public-gw]
连接顺序：
  1. 先直连 public-gw（公网跳板机）
  2. 通过 public-gw 打开 direct-tcpip 到 dmz-relay
  3. 通过 dmz-relay 打开 direct-tcpip 到 internal-db
  4. 建立 internal-db 的 SSH 会话
```

**防循环检测**：解析时用 HashSet 检测，若 server_id 重复出现则报错。

### 3. SSH Agent Forwarding

新增认证方式 `AuthType::Agent`，使用系统 SSH Agent（`$SSH_AUTH_SOCK`）完成认证，私钥无需存储在 Termex 内：

```rust
pub enum AuthType {
    Password,
    Key,
    Agent,  // 新增：使用本机 SSH Agent
}
```

**适用场景**：
- 开发者已配置本机 SSH Agent
- 私钥存储于 YubiKey / Hardware Token，不适合复制
- 多跳板机场景，避免私钥泄露风险

### 4. ConnectModal Tab 化重构

现有单页表单拆分为固定区域 + 两个 Tab：

```
┌─ 固定区域 ─────────────────────────────┐
│ 名称  [________________]               │
│ Host  [______________] Port [22]       │
│ 用户名 [________________]               │
├─ Tab 1: 授权信息  │ Tab 2: SSH 隧道 ──┤
│                                        │
│ 认证方式：                              │
│  ● 密码  ○ 密钥  ○ SSH Agent          │
│                                        │
│ [对应字段...]                          │
│                                        │
│ 分组  [v] 选择分组...                  │
└────────────────────────────────────────┘
```

**Tab 1：授权信息**
- 认证方式（密码 / 密钥 / SSH Agent）
- 根据选择显示对应字段
- 分组选择器

**Tab 2：SSH 隧道**
- 跳板机选择器（filterable + clearable）
- 连接链路预览（动态渲染）
- 未配置时显示「将直接连接」提示

## 架构亮点

1. **连接池而非嵌套** — 多个内网服务器共享跳板机 Session，避免连接爆炸
2. **引用计数自动管理** — ref_count 降为 0 时自动关闭，无资源泄露
3. **多级支持无成本** — 递归解析 + 倒序连接，任意层数无特殊处理
4. **Agent 私钥不落地** — 跳板机、内网服务器均可用 Agent 认证，闭环无漏
5. **Sidebar 即时反馈** — 跳板机变更实时影响 UI，引用计数自动更新

## 技术风险 & 应对

| 风险 | 应对 |
|---|---|
| russh `connect_stream()` channel.into_stream() 兼容性 | Phase 1 先写 PoC，验证 russh 0.46 支持 |
| 连接池并发安全 | TokioRwLock + 引用计数在写锁内操作 |
| 跳板机断线导致内网 session 异常 | 捕获错误，emit ssh://status error event |
| SSH_AUTH_SOCK 跨平台（Windows）| Pageant / OpenSSH Agent，启动时检测 |
| 循环代理死锁 | HashSet 解析时检测，报错不建立连接 |
| 多级跳板连接顺序混乱 | iter().rev() 确保倒序，单元测试覆盖 |

## 设计决策（已确认）

| 决策 | 方案 | 原因 |
|------|------|------|
| 连接池架构 | 外部连接池（不嵌套） | 支持多级跳板、Session 复用、生命周期清晰 |
| 多级跳板支持 | 一步到位，递归解析 | 生产场景常见，处理连接顺序即可支持 |
| SSH Agent | 完整支持 | 私钥安全，跳板机 + 内网都可用 |
| 内网 server 标识 | 方案 A + B 都做 | 跳板机显示徽章，内网显示箭头，信息完整 |
| Session 复用 | 共享，不限引用数 | 资源高效，1 个跳板机支持无限内网 server |

## 预期效果

**之前**：
```
用户：我要连接内网 DB（172.16.1.10）
解决方案：
  1. 先连接公网跳板机（手动）
  2. 再在跳板机上 SSH 到内网 DB（手动）
  3. 两个独立的终端窗口，体验割裂
```

**之后**：
```
用户：我要连接内网 DB
解决方案：
  1. 配置一次跳板机（bastion-01）
  2. 内网 DB 配置 proxy_id=bastion-01
  3. 点击连接，一切自动化

结果：
  - 对用户完全透明（跳板中转自动进行）
  - 跳板机变更一次配置即可生效（所有依赖自动更新）
  - Sidebar 即时反馈（徽章、引用数、链路预览）
  - 支持多级跳板、SSH Agent 无缝集成
```

这是运维场景的生产级解决方案。
