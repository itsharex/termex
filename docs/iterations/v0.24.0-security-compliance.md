# V0.24.0 — 安全合规改造 (GDPR / ISO 27001 / 等保 2.0)

## 背景

Termex 作为 SSH 客户端，处理高敏感数据：服务器凭据、私钥、终端会话内容。随着软件成熟度提升，需要针对三大合规框架进行全面安全评估与改造：

- **GDPR** (EU General Data Protection Regulation) — 用户数据保护、知情权、删除权
- **ISO 27001 / ISO 27002** — 信息安全管理体系 (ISMS)，访问控制、密码学、通信安全
- **中国等保 2.0** (GB/T 22239-2019) — 网络安全等级保护，身份鉴别、访问控制、数据完整性、审计

### 当前安全评估摘要

| 领域 | 现状 | 风险等级 |
|------|------|---------|
| SSH Host Key 验证 | `check_server_key()` 无条件返回 `true` | **严重** |
| CSP (Content Security Policy) | `tauri.conf.json` 中 `csp: null`（已禁用） | **高** |
| 数据库加密 | `AppState::new(None)` — 默认不加密 | **高** |
| 内存凭据清零 | 仅 `master_password_lock` 手动清零，无 `zeroize` | **高** |
| 安全审计日志 | 无 — 无登录/操作/失败审计 | **高** |
| 密码强度策略 | 主密码无最低要求 | **中** |
| 会话超时锁定 | 无自动锁定机制 | **中** |
| 隐私声明/用户同意 | 菜单有 Privacy Policy 但未实现内容 | **中** |
| 数据导出/删除权 | 有导出功能，无一键清除 | **中** |
| 连接超时/重试限制 | SSH 硬编码 10s 超时，无失败锁定 | **低** |

## 一、SSH Host Key 验证 (TOFU) — 严重

### 1.1 问题

[auth.rs:49-54](src-tauri/src/ssh/auth.rs#L49-L54) 中 `check_server_key()` 无条件接受所有服务器公钥，完全无法防御中间人攻击 (MITM)。

**违反**: ISO 27001 A.10.1 (密码学控制)、等保 2.0 身份鉴别 (8.1.4.1)

### 1.2 改造方案

采用 Trust On First Use (TOFU) 模型，与 OpenSSH 行为一致：

**新建文件**: `src-tauri/src/ssh/host_key.rs`

```rust
pub enum HostKeyVerifyResult {
    Trusted,          // 指纹匹配 known_hosts
    NewHost,          // 首次连接，需用户确认
    KeyChanged,       // 指纹变化 — 可能 MITM，需强制确认
}

/// 验证服务器公钥（TOFU 模型）
pub fn verify_host_key(
    db: &Database,
    host: &str,
    port: u16,
    server_public_key: &PublicKey,
) -> Result<HostKeyVerifyResult, SshError>

/// 存储已信任的主机指纹
pub fn trust_host_key(
    db: &Database,
    host: &str,
    port: u16,
    server_public_key: &PublicKey,
) -> Result<(), SshError>
```

**修改文件**: `src-tauri/src/ssh/auth.rs`

- `ClientHandler` 增加 `host_key_tx: Option<oneshot::Sender<PublicKey>>` 字段
- `check_server_key()` 将公钥发送给调用方，由 `ssh_connect` 命令判断是否信任
- 首次连接或密钥变更时，通过 Tauri event 通知前端弹出确认对话框

**修改文件**: `src-tauri/src/commands/ssh.rs`

- `ssh_connect` 中加入 host key 验证流程
- `KeyChanged` 时 emit `ssh://host-key-changed/{session_id}` 到前端
- 新增 `ssh_trust_host_key` 命令用于用户确认后存储指纹

**新建前端组件**: `src/components/terminal/HostKeyDialog.vue`

- 展示主机名、端口、指纹类型、指纹值
- NewHost: "首次连接此服务器，是否信任？"
- KeyChanged: **红色警告** "服务器密钥已变更！这可能意味着中间人攻击。"

### 1.3 数据库

已有 `known_hosts` 表 (migration v1)，包含 `host, port, key_type, fingerprint, first_seen, last_seen` 字段，无需新增 migration。

## 二、Content Security Policy — 高

### 2.1 问题

[tauri.conf.json:28](src-tauri/tauri.conf.json#L28) 中 `"csp": null` 完全禁用了 CSP，允许任意脚本注入和资源加载。

**违反**: ISO 27001 A.14.2 (安全开发)、OWASP Top 10 (XSS)

### 2.2 改造方案

**修改文件**: `src-tauri/tauri.conf.json`

```json
{
  "app": {
    "security": {
      "csp": "default-src 'self'; script-src 'self' 'wasm-unsafe-eval'; style-src 'self' 'unsafe-inline'; font-src 'self' data:; connect-src 'self' https:; img-src 'self' data: blob:"
    }
  }
}
```

策略说明：
- `default-src 'self'`: 默认只允许同源资源
- `script-src 'self' 'wasm-unsafe-eval'`: 允许 xterm.js WebGL renderer 的 WASM
- `style-src 'self' 'unsafe-inline'`: Element Plus 需要内联样式
- `connect-src 'self' https:`: AI provider API 调用需要 HTTPS 出站
- `img-src 'self' data: blob:`: 允许内联图片 (base64 icons)
- 禁止 `eval()`、禁止 HTTP 资源加载、禁止 iframe 嵌入

## 三、数据库强制加密 — 高

### 3.1 问题

[lib.rs:158](src-tauri/src/lib.rs#L158) 中 `AppState::new(None)` 默认以无密码模式打开数据库，所有数据明文存储。

**违反**: GDPR Art.32 (数据安全技术措施)、等保 2.0 数据保密性 (8.1.4.4)、ISO 27001 A.10.1

### 3.2 改造方案

**引导式首次启动加密**：

1. 首次启动自动生成随机 DB 密钥，存入 OS Keychain（`termex:db:master`）
2. 无需用户手动设置密码即可实现静态加密
3. 用户仍可设置主密码（Master Password）作为第二层保护（加密凭据字段）
4. Portable 模式：强制要求设置主密码（无 Keychain 可用时的唯一加密途径）

**修改文件**: `src-tauri/src/lib.rs`

```rust
// 替代 AppState::new(None)
let db_key = keychain::get("termex:db:master")
    .unwrap_or_else(|_| {
        let key = generate_random_db_key();
        keychain::store("termex:db:master", &key).ok();
        key
    });
let app_state = AppState::new(Some(&db_key))?;
```

**修改文件**: `src-tauri/src/storage/db.rs`

- 添加 DB 加密迁移逻辑：检测现有未加密 DB → 备份 → 导出 → 加密重建
- 使用 SQLCipher `PRAGMA rekey` 实现在线加密

## 四、内存凭据安全 (zeroize) — 高

### 4.1 问题

密码、密钥在内存中以普通 `String` / `[u8; 32]` 存在，仅 `master_password_lock` 时手动 `fill(0)`，其余路径（SSH 认证、KDF 派生、解密中间值）均未清零。

**违反**: ISO 27001 A.10.1 (密码学控制)、等保 2.0 数据保密性

### 4.2 改造方案

**新增依赖**: `zeroize = { version = "1", features = ["zeroize_derive"] }`

**修改文件列表**:

| 文件 | 改动 |
|------|------|
| `src-tauri/src/crypto/kdf.rs` | `derive_key()` 返回 `Zeroizing<[u8; 32]>`，密码参数使用后自动清零 |
| `src-tauri/src/crypto/aes.rs` | 加解密中间 buffer 使用 `Zeroizing<Vec<u8>>` |
| `src-tauri/src/state.rs` | `master_key: RwLock<Option<Zeroizing<[u8; 32]>>>` |
| `src-tauri/src/commands/crypto.rs` | 所有 `password: String` 参数解密/使用后 zeroize |
| `src-tauri/src/commands/ssh.rs` | SSH 密码从 keychain 读取后，认证完成立即清零 |
| `src-tauri/src/keychain/mod.rs` | `get()` 返回 `Zeroizing<String>`，调用方使用后自动清零 |

关键原则：
- 所有密钥材料使用 `Zeroizing<T>` 包装，Drop 时自动清零
- 禁止密钥/密码的 `.clone()` — 使用引用传递
- KDF 输入密码用完立即清零

## 七、密码强度策略 — 中

### 7.1 问题

主密码无最低强度要求，用户可设置 "1" 等弱密码。

**违反**: 等保 2.0 身份鉴别 (8.1.4.1 口令复杂度)、ISO 27001 A.9.4 (密码管理)

### 7.2 改造方案

**新建文件**: `src-tauri/src/crypto/password_policy.rs`

```rust
pub struct PasswordStrength {
    pub score: u8,          // 0-4 (zxcvbn 风格)
    pub feedback: Vec<String>,
}

pub fn check_strength(password: &str) -> PasswordStrength {
    // 规则：
    // - 最低 8 字符
    // - 不在常见密码黑名单 (top 10000)
    // - 包含大小写 + 数字（等保 2.0 要求）
    // - 不包含用户名或 "termex" 等上下文词
}

pub fn validate_master_password(password: &str) -> Result<(), Vec<String>> {
    let strength = check_strength(password);
    if strength.score < 2 {
        return Err(strength.feedback);
    }
    Ok(())
}
```

**修改文件**: `src-tauri/src/commands/crypto.rs`

- `master_password_set` 和 `master_password_change` 中调用 `validate_master_password()`

**前端**: 密码设置/修改 UI 增加实时强度指示器 (弱/中/强/极强)

## 八、认证失败锁定 — 中

### 8.1 问题

**违反**: 等保 2.0 身份鉴别 (8.1.4.1 登录失败处理)

### 8.2 改造方案

**修改文件**: `src-tauri/src/commands/crypto.rs`

```rust
// 内存计数器 (不需要持久化 — 重启即重置)
static FAILED_ATTEMPTS: AtomicU32 = AtomicU32::new(0);

// master_password_verify 中：
// - 失败次数 < 5: 正常返回
// - 5 次后: 强制等待 30 秒
// - 10 次后: 强制等待 5 分钟
// - 每次成功: 重置计数
```

**前端**: 显示剩余等待时间倒计时

## 十、隐私声明与用户同意 — 中

### 10.1 问题

菜单栏有 "Privacy Policy" 入口但未实现内容。GDPR 要求在收集/处理数据前告知用户。

### 10.2 改造方案

**新建文件**: `src/components/PrivacyPolicy.vue`

内容要点（面向 Termex 的特殊性）：
- **本地存储**: 所有数据仅存储在本机，不上传任何服务器
- **AI 功能**: 启用 AI 时，命令文本发送给第三方 AI 提供商（用户自行配置的 API）
- **不发送**: 密码、密钥、服务器 IP 不会发送给 AI
- **审计日志**: 仅本地存储，不外传
- **OS Keychain**: 使用系统凭据存储机制
- **遥测**: 无遥测、无分析、无数据回传

**首次启动**: 显示隐私声明摘要，用户确认后写入 settings `privacy_accepted_at`。

## 十一、Tauri IPC 安全加固 — 低

### 11.1 当前状态

[capabilities/default.json](src-tauri/capabilities/default.json) 仅定义了窗口管理权限。Tauri v2 的 IPC 命令默认全部可用（不像 Tauri v1 需要白名单）。

### 11.2 改造方案

无需改动 — Tauri v2 的 `invoke_handler` 注册机制本身就是白名单模式，未注册的命令无法调用。当前架构已符合最小权限原则。

## 十二、SSH 连接安全增强 — 低

### 12.1 改造项

**修改文件**: `src-tauri/src/ssh/session.rs`

1. **禁用弱加密算法**: 配置 russh 只使用强加密套件
   - 密钥交换: curve25519-sha256, ecdh-sha2-nistp256
   - 加密: chacha20-poly1305, aes256-gcm, aes256-ctr
   - MAC: hmac-sha2-256, hmac-sha2-512
   - 禁用: arcfour, 3des-cbc, hmac-md5, hmac-sha1

2. **SSH Banner 信息收敛**: 不在错误信息中暴露 SSH 协议细节
